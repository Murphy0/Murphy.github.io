Mysql

数据库 mysql
数据库
发布日期:   2020-12-26

1、介绍

​ MySQL 是一种关系型数据库，默认端口号是3306。

2、Explain

  • id：表示查询中执行select子句或操作表的顺序。

  • select_type：查询类型。

  • type：表示msql找到数据的方式 。

    system, const, eq_ref, ref, fulltext, ref_or_null, index_merge, unique_subquery, index_subquery, range, index, ALL

  • possible_keys：MySQL查找表中的行时可以选择的索引。

  • key：显示MySQL在查询中实际决定使用的索引，若没有使用索引，显示为NULL。

  • key_len：表示MySQL决定使用的键的长度（字节数）。

  • ref

  • rows：表示MySQL认为执行该查询必须检查的行数（是一个估计值）。

  • filtered：该列表示将被表条件过滤的表行的估计百分比。

  • Extra：附加信息。

    • Using index：该值表示相应的select操作中使用了覆盖索引
    • Using where：表示mysql服务器将在存储引擎检索行后再进行过滤。
    • Using temporary：表示MySQL需要使用临时表来存储结果集，常见于排序和分组查询。出现此种说明sql可能需要优化
    • Using filesort：MySQL中无法利用索引完成的排序操作称为“文件排序”。出现此种说明sql可能需要优化
    • Using join buffer：该值强调了在获取连接条件时没有使用索引，并且需要连接缓冲区来存储中间结果。
    • Impossible where：这个值强调了where语句会导致没有符合条件的行。
    • Select tables optimized away
    • Index merges

3、InnoDB

  • 支持行级锁和表级锁
  • 当操作未走到索引的时候走的是表级锁，当操作走到索引的时候使用行级锁

4、B+Tree

  • 非叶子节点的子树指针与关键字相同
  • 非叶子节点仅用来存储索引，数据都保存在叶子节点中
  • 所有叶子节点均有一个链指针指向下一个叶子节点

5、数据库锁分类

  • 锁粒度划分

    • 表级
    • 行级
    • 页级

  • 锁级别划分

    • 共享锁（读锁）
    • 排它锁（写锁）

  • 加锁方式划分

    • 自动锁（即执行update 等语句）
    • 显式 锁

  • 操作划分

    • DML锁
    • DDL锁

  • 使用方式划分

    • 乐观锁

      • 其表示数据一般不会冲突，故只有在数据提交的时候才会加锁检测，一般其实现方式有2种：

        （1）版本号

        （2）时间戳

    • 悲观锁

      • 保证对事务的排它性，全程使用排它锁是其一种实现

6、数据事务四大特性(ACID)

  • A（Atomic ）:原子性
    • 事务包含的操作要么全部成功，要么全部失败
  • C（Consistency）:一致性
    • 数据应满足完整性约束，数据库应确保数据的状态从一个一致状态转成另一个一致状态，如用户A和用户B总共有1000元，那么他们之间不论如何操作，总金额始终是1000元
  • I（Isolation）:隔离性
    • 一个事务的执行不影响其他事务的执行
  • D(Durability):持久性
    • 事务一旦提交，数据即已更新，其实现在于事务一旦提交会更新相关系统文件，一旦服务重启也能恢复数据

7、事务隔离级别

  • 读未提交
    • 存在脏读、不可重复读、幻读问题
  • 读已提交（READ-COMMITTED），oracal默认
    • 存在不可重复读、幻读问题
  • 可重复读（REPEATABLE-READ），mysql默认
    • 存在幻读问题
  • 串行化（SERIALIZABLE）

8、当前读、快照读

  • 当前读：指加锁的操作，即当前操作的就是最新版本数据
  • 快照读：指未加锁的操作，如select 也就是说本次操作不一定是最新版本

9、Gap锁（间隙锁）

​ gap锁是用来防止插入，避免幻读发生，只会在gap锁的左开右闭的区间生效

10、聚集索引和非聚集索引

  • 在 InnoDB 里，如果存在主键，则其为密集索引，如果不存在主键，则第一个唯一非空索引为密集索引 ，否则InnoDB就会生成一个隐藏主键
  • 非主键索引被称为辅助索引。非主键索引的叶子节点不存储数据，而是存储的主键值，查询到主键数据后，需要再根据主键索引二次定位到具体的数据

10.1、区别：

  • 聚集索引一个表只能有一个，而非聚集索引一个表可以存在多个
  • 聚集索引存储记录是物理上连续存在，而非聚集索引是逻辑上的连续，物理存储并不连续。

11、binlog、redo log、undo log

  • redo log是InnoDB引擎特有的；binlog是MySQL的Server层实现的，所有引擎都可以使用。
  • redo log是物理日志，记录的是“在某个数据页上做了什么修改”；binlog是逻辑日志，记录的是这个语句的原始逻辑
  • redo log是循环写的，空间固定会用完；binlog是可以追加写入的。“追加写”是指binlog文件写到一定大小后会切换到下一个，并不会覆盖以前的日志。

1、binlog（二进制日志）

  • 用于复制，在主从复制中，从库利用主库上的binlog进行重播，实现主从同步。
  • 用于数据库的基于时间点的还原。

2、undo log（回滚日志）

  • 保存了事务发生之前的数据的一个版本，可以用于回滚

3、redo log（重做日志）

  • 确保事务的持久性。防止在发生故障的时间点，尚有脏页未写入磁盘，在重启mysql服务的时候，根据redo log进行重做，从而达到事务的持久性这一特性。

12、InnoDB、MyISAM

1、区别

  • InnoDB支持事务（TRANSACTION）
  • InnoDB 最小的锁粒度是行锁，MyISAM 最小的锁粒度是表锁。一个更新语句会锁住整张表，导致其他查询和更新都会被阻塞，因此并发访问受限。
  • InnoDB 支持外键，而 MyISAM 不支持。
  • InnoDB 是聚集索引，MyISAM 是非聚集索引。
  • InnoDB 不保存表的具体行数，执行 select count(*) from table 时需要全表扫描。而MyISAM 用一个变量保存了整个表的行数，执行上述语句时只需要读出该变量即可，速度很快；

2、相关锁比较

  • InnoDB默认select 操作不加锁，同时默认当不走索引的时候加表锁，走索引的时候加行锁手动在sql后添加lock in share mode 表示使用共享锁

  • MyISAM默认select 操作加读锁（共享锁），更新操作加锁（排他锁）

  • InnoDB行锁是通过给索引上的索引项加锁来实现的， InnoDB这种行锁实现特点意味着：只有通过索引条件检索数据，InnoDB才使用行级锁，否则，InnoDB将使用表锁！

  • 由于MySQL的行锁是针对索引加的锁，不是针对记录加的锁，所以虽然是访问不同行的记录，但是如果是使用相同的索引键，是会出现锁冲突的。**
